백준 2042번 **구간 합 구하기** 문제는 배열의 값이 빈번하게 바뀌는 상황에서 특정 구간의 합을 빠르게 구해야 하는 문제입니다.

이 문제는 일반적인 누적 합(Prefix Sum)이나 단순 반복문으로 풀면 **무조건 시간 초과**가 발생합니다. 데이터의 변경과 구간 합 계산을 모두 $O(\log N)$만에 처리할 수 있는 **세그먼트 트리(Segment Tree)** 또는 **펜윅 트리(Fenwick Tree, BIT)**를 사용해야 하는 가장 대표적인 문제입니다.

여기서는 직관적이고 활용도가 가장 높은 **세그먼트 트리**를 기준으로 설명해 드릴게요.

### 💡 핵심 아이디어: 세그먼트 트리(Segment Tree)

1. **왜 단순 배열은 안 될까?**
* 값이 바뀔 때마다 누적 합 배열을 다시 만들거나, 매번 반복문으로 합을 구하면 1번 연산에 최대 $O(N)$의 시간이 걸립니다.
* $N$이 $10^6$이고 변경/출력 쿼리가 $2 \times 10^4$번이면 약 200억 번의 연산이 필요해 시간 초과가 납니다.


2. **세그먼트 트리란?**
* 트리의 각 노드가 **"특정 구간의 합"**을 저장하고 있는 자료구조입니다.
* 루트 노드는 전체 구간(1~N)의 합을, 그 자식들은 절반씩 나눈 구간의 합을 가집니다.
* 값을 변경하거나 구간 합을 구할 때, 트리의 높이만큼만 탐색하면 되므로 **$O(\log N)$**이라는 압도적인 속도를 자랑합니다.

### 1. Java 풀이

숫자의 범위가 매우 크므로 합을 담는 변수들은 반드시 **`long`** 타입을 사용해야 합니다. 세그먼트 트리의 크기는 넉넉하게 배열 크기 $N$의 4배로 잡는 것이 정석입니다.

```java
import java.io.*;
import java.util.*;

public class Main {
    static long[] arr;
    static long[] tree;

    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        StringTokenizer st = new StringTokenizer(br.readLine());

        int N = Integer.parseInt(st.nextToken());
        int M = Integer.parseInt(st.nextToken()); // 변경 일어나는 횟수
        int K = Integer.parseInt(st.nextToken()); // 구간 합 구하는 횟수

        arr = new long[N + 1];
        tree = new long[N * 4]; // 트리의 크기는 N의 4배면 모든 노드를 수용 가능

        for (int i = 1; i <= N; i++) {
            arr[i] = Long.parseLong(br.readLine());
        }

        // 1. 세그먼트 트리 생성
        build(1, 1, N);

        StringBuilder sb = new StringBuilder();
        for (int i = 0; i < M + K; i++) {
            st = new StringTokenizer(br.readLine());
            int a = Integer.parseInt(st.nextToken());
            int b = Integer.parseInt(st.nextToken());
            long c = Long.parseLong(st.nextToken());

            if (a == 1) {
                // b번째 수를 c로 변경
                // 기존 값과 새 값의 차이(diff)를 구해서 트리 업데이트
                long diff = c - arr[b];
                arr[b] = c; // 원본 배열 업데이트
                update(1, 1, N, b, diff);
            } else if (a == 2) {
                // b부터 c까지의 구간 합 출력
                sb.append(sum(1, 1, N, b, (int) c)).append("\n");
            }
        }
        System.out.print(sb);
    }

    // 트리 초기화 (node: 현재 노드 번호, start~end: 노드가 담당하는 원본 배열의 구간)
    static long build(int node, int start, int end) {
        if (start == end) { // 리프 노드 (구간의 길이가 1)
            return tree[node] = arr[start];
        }
        int mid = (start + end) / 2;
        // 왼쪽 자식과 오른쪽 자식의 합을 현재 노드에 저장
        return tree[node] = build(node * 2, start, mid) + build(node * 2 + 1, mid + 1, end);
    }

    // 값 변경 (index: 변경할 원본 배열의 인덱스, diff: 변경된 차이값)
    static void update(int node, int start, int end, int index, long diff) {
        // 변경할 인덱스가 현재 노드의 담당 구간을 벗어나면 무시
        if (index < start || index > end) return;

        // 구간에 포함된다면 차이(diff)만큼 더해줌
        tree[node] += diff;

        // 리프 노드가 아니라면 자식들도 업데이트
        if (start != end) {
            int mid = (start + end) / 2;
            update(node * 2, start, mid, index, diff);
            update(node * 2 + 1, mid + 1, end, index, diff);
        }
    }

    // 구간 합 구하기 (left~right: 구하고자 하는 구간 합의 범위)
    static long sum(int node, int start, int end, int left, int right) {
        // 1. 구하려는 범위가 노드의 담당 구간을 완전히 벗어난 경우
        if (left > end || right < start) return 0;

        // 2. 구하려는 범위가 노드의 담당 구간을 완전히 포함하는 경우 (더 이상 안 내려가고 바로 리턴)
        if (left <= start && end <= right) return tree[node];

        // 3. 그 외의 경우 (걸쳐 있는 경우) -> 자식 노드들로 더 파고들어감
        int mid = (start + end) / 2;
        return sum(node * 2, start, mid, left, right) + sum(node * 2 + 1, mid + 1, end, left, right);
    }
}

```

### 2. Python 풀이

파이썬은 재귀 깊이 제한이 기본적으로 낮게 설정되어 있으므로, 트리를 타고 내려가는 세그먼트 트리를 구현할 때는 `sys.setrecursionlimit`을 늘려주어야 안전합니다.

```python
import sys
sys.setrecursionlimit(10**5)
input = sys.stdin.readline

def build(node, start, end):
    if start == end:
        tree[node] = arr[start]
        return tree[node]
    
    mid = (start + end) // 2
    tree[node] = build(node * 2, start, mid) + build(node * 2 + 1, mid + 1, end)
    return tree[node]

def update(node, start, end, index, diff):
    if index < start or index > end:
        return
    
    tree[node] += diff
    
    if start != end:
        mid = (start + end) // 2
        update(node * 2, start, mid, index, diff)
        update(node * 2 + 1, mid + 1, end, index, diff)

def query(node, start, end, left, right):
    # 범위 밖
    if left > end or right < start:
        return 0
    # 범위 안에 완전히 포함
    if left <= start and end <= right:
        return tree[node]
    
    # 일부만 걸친 경우
    mid = (start + end) // 2
    return query(node * 2, start, mid, left, right) + query(node * 2 + 1, mid + 1, end, left, right)

def solve():
    global arr, tree
    
    N, M, K = map(int, input().split())
    
    arr = [0] * (N + 1)
    tree = [0] * (N * 4)
    
    for i in range(1, N + 1):
        arr[i] = int(input())
        
    build(1, 1, N)
    
    for _ in range(M + K):
        a, b, c = map(int, input().split())
        if a == 1:
            diff = c - arr[b]
            arr[b] = c
            update(1, 1, N, b, diff)
        elif a == 2:
            print(query(1, 1, N, b, c))

if __name__ == "__main__":
    solve()

```

### 🔍 풀이 포인트

1. **`diff`를 이용한 업데이트:** 배열의 특정 값을 통째로 바꾸는 대신, **"새로운 값 - 기존 값"**의 차이(`diff`)를 구합니다. 그리고 그 인덱스(`b`)를 포함하고 있는 세그먼트 트리의 모든 상위 노드들에 이 `diff`를 더해주면 아주 효율적으로 트리 전체가 갱신됩니다.
2. **트리 배열의 크기:** $N$개의 리프 노드를 가지는 포화 이진 트리(Perfect Binary Tree)를 배열로 구현할 때, 트리의 높이를 계산해서 메모리를 딱 맞게 할당할 수도 있지만 실전에서는 단순히 **`N * 4`** 사이즈로 배열을 잡으면 오버플로우 없이 넉넉하게 모든 노드를 담을 수 있습니다.

세그먼트 트리는 처음 접하면 재귀 호출 때문에 살짝 복잡하게 느껴질 수 있습니다. `build`, `update`, `sum` 세 가지 함수 중 흐름이 잘 안 그려지는 부분이 있다면 편하게 말씀해주세요!

왜 배열의 크리가 N*4인가

이 질문은 세그먼트 트리를 공부할 때 누구나 한 번쯤 품게 되는 아주 훌륭한 수학적 의문입니다!

왜 하필 **`N * 4`**일까요? 결론부터 말씀드리면, 세그먼트 트리를 배열로 구현할 때 **"트리의 빈 공간(안 쓰는 인덱스)"까지 모두 고려한 최악의 경우(Worst Case)**를 안전하게 덮기 위해서입니다.

이해하기 쉽게 두 가지 케이스로 나누어 설명해 드릴게요.

---

### 1. 이상적인 케이스 (N이 2의 거듭제곱일 때)

배열의 원소 개수 $N$이 **8**($=2^3$)이라고 가정해 봅시다.
세그먼트 트리는 데이터를 반씩 쪼개어 내려가는 완벽한 '포화 이진 트리(Perfect Binary Tree)' 모양이 됩니다.

* **맨 아래 리프 노드 (원본 데이터):** 8개
* **그 위를 묶어주는 내부 노드 (구간 합):** 7개 ($4 + 2 + 1$)
* **총 필요한 노드 개수:** 15개 ($2N - 1$)

이 경우, 배열의 인덱스는 1번부터 15번까지만 사용되므로 **`N * 2`** 크기만 잡아도 아주 넉넉합니다.

### 2. 최악의 케이스 (N이 2의 거듭제곱이 아닐 때)

문제는 $N$이 **9**일 때입니다. 8보다 딱 1개 많아졌을 뿐인데 트리의 구조는 어떻게 변할까요?

기존의 8개짜리 트리에 9번째 데이터를 넣으려면, 트리의 층수(높이)가 한 층 더 밑으로 깊어져야 합니다. 이진 트리의 특성상 한 층이 깊어지면, 사실상 **$N=16$ ($=2^4$)인 완벽한 트리를 만들고 그중 일부를 비워두는(안 쓰는) 형태**가 되어버립니다.

* $N=9$일 때, 알고리즘은 이를 품기 위해 그보다 큰 '가장 가까운 2의 거듭제곱'인 **16**을 기준으로 트리를 구성합니다.
* 이때 배열의 최대 인덱스는 $16 \times 2 - 1 = 31$번까지 튀게 됩니다.
* $N=9$인데 인덱스가 31까지 필요하니, `N * 2` (18)로는 어림도 없고 `IndexOutOfBoundsException` 에러가 터지게 됩니다.

### 💡 수학적 결론: 왜 4배인가?

수학적으로 정리하면 이렇습니다. $N$보다 크거나 같은 가장 작은 2의 거듭제곱을 $K$라고 해봅시다. 필요한 최대 배열 크기는 **$2K$**입니다.

최악의 상황은 $N$이 2의 거듭제곱보다 아주 살짝 클 때 (예: $N=9$) 발생합니다. 이때 $K$는 $N$의 거의 2배에 가까워집니다 ($K \approx 2N$).

따라서 필요한 노드의 총 개수를 계산하는 공식은 다음과 같이 전개됩니다.


$$\text{필요한 최대 크기} \approx 2 \times K < 2 \times (2N) = 4N$$

즉, 아무리 트리가 비효율적으로 빈 공간을 낭비하며 뻗어나가더라도, 그 인덱스는 **$4N$을 절대 넘을 수 없다**는 것이 수학적으로 보장됩니다.

그래서 실전 코딩 테스트에서는 귀찮게 `Math.ceil(Math.log(N) / Math.log(2))` 같은 계산을 해서 정확한 크기를 구하는 대신, 쿨하게 **`N * 4`**를 곱해서 오버플로우 걱정 없이 안전하게 배열을 선언하는 것이 일종의 '국룰'이 된 것입니다! 궁금증이 완벽하게 풀리셨나요?
